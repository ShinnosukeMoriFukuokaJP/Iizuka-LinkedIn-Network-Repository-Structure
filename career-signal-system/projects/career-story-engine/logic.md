# Career Story Engine — Logic Reference

## The PSI Transform

### P — Problem

**Bad (common mistake):**
> "I wanted to learn Python so I built a script."

**Good (PSI format):**
> "Manual data reformatting between LinkedIn and GitHub created
> inconsistency and consumed 30+ minutes per update cycle."

**Rule:** The problem must be felt by a real person (including yourself).

---

### S — Solution

**Bad:**
> "I used Python with the GitHub API and urllib."

**Good:**
> "A lightweight pipeline that fetches existing file SHA,
> encodes content, and pushes via GitHub Contents API —
> no external dependencies required."

**Rule:** Describe the *structure* of your solution, not just the tools.

---

### I — Impact

**Bad (inflated):**
> "Reduced manual work by 99%"

**Bad (too vague):**
> "Made things better"

**Good (structural, honest):**
> "Converted a 45-minute manual update process into a single
> Python script execution — repeatable and version-controlled."

**Good (no numbers, still valid):**
> "Established a consistent documentation structure across
> all projects, eliminating per-project formatting decisions."

**Rule:** If you have numbers, use them. If not, describe the *structural change*.

---

## Impact Type Matrix

| Type | When to use | Example |
|------|------------|---------|
| Time reduction | Automation replaced manual steps | "15-min task → 30-sec script" |
| Structural | Process went from ad-hoc to systematic | "Ad-hoc → reproducible framework" |
| Clarity | Ambiguity was eliminated | "6 separate docs → 1 source of truth" |
| Reproducibility | Others can repeat it | "Reusable template for any project" |
| Scale | Same effort covers more | "1 script → handles N projects" |

---

## For Early-Stage Developers

You do not need:
- ❌ Years of experience
- ❌ Large team projects
- ❌ Hard metrics on every project

You do need:
- ✓ A real problem (even a personal one)
- ✓ Something you actually built
- ✓ An honest description of what changed

> Structural honesty beats resume inflation every time.
