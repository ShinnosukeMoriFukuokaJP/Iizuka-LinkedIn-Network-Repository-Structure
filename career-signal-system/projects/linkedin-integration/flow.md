# LinkedIn ↔ GitHub Integration Flow

## Outbound Flow (LinkedIn → GitHub)

```
1. LinkedIn Post published
   ↓
2. Post includes GitHub project link
   ↓
3. Viewer lands on GitHub README
   ↓
4. README links to Case Study
   ↓
5. Case Study demonstrates Impact
   ↓
6. Viewer returns to LinkedIn with context
```

## Inbound Flow (GitHub → LinkedIn)

```
1. Developer completes a project
   ↓
2. Writes README (Problem / Solution / Impact)
   ↓
3. Publishes LinkedIn post about the project
   ↓
4. Links Featured section to GitHub
   ↓
5. Engagement signals trigger LinkedIn algorithm
   ↓
6. New viewers enter at LinkedIn and cycle begins
```

## Maintenance Cadence

| Action | Frequency |
|--------|-----------|
| LinkedIn post (project update) | Weekly |
| README update | Per project milestone |
| Case study addition | Monthly |
| Featured section refresh | Quarterly |

## Key Principle

The goal is not **more content**.  
The goal is **connected content** — each piece links to the next.

> A single well-connected project outperforms ten isolated ones.
