## 問題

LinkedInとGitHubに分散したキャリア情報は、採用担当者に**30秒以内に評価されない**。  
技術的な実力は存在するのに、それが正しく伝わらない。  
原因はスキルの不足ではなく、**シグナル設計の不足**だ。

## 解決策

LinkedInのナレーティブ、GitHubの実装、ケーススタディを  
**一つの読みやすいキャリアシグナル**に統合するシステム。

このリポジトリそのものが、そのプロセスの証明です。

## このシステムが行うこと

- プロジェクトを **問題 → 解決 → インパクト（PSI）** 構造に整理する
- 技術的な実装を**ビジネスで読める価値**に変換する
- LinkedInとGitHub全体で一貫したプロフェッショナルな存在感を構築する
- 実績を誇張せずに、**採用担当者の評価スピードを上げる**

## インパクト比較

| 評価軸 | 導入前 | 導入後 |
|--------|--------|--------|
| アイデンティティの明確さ | プラットフォームをまたいで分散 | 統合されたシグナルレイヤー |
| プロジェクトの読みやすさ | コード優先・文脈なし | 価値優先・ストーリー付き |
| 採用担当者の評価 | 高い摩擦 | 低い摩擦 |
| 再現性 | 場当たり的 | 構造化されたフレームワーク |

## システムアーキテクチャ

```
LinkedIn（ナレーティブ層）
    ↓  投稿・記事・Featured
GitHub（実行層）
    ↓  プロジェクト・README・スクリプト
ケーススタディ（インパクト層）
    ↓  成果の記録
    ↺  LinkedInの更新投稿へ戻る
```

## PSI構造とは

このシステムのすべてのドキュメントは **Problem → Solution → Impact** で書かれています。

| レイヤー | 問い | 出力 |
|---------|------|------|
| Problem（問題） | 何が壊れていたか？ | 1文の問題定義 |
| Solution（解決） | 何を作ったか？ | システム説明（専門用語なし） |
| Impact（インパクト） | 何が変わったか？ | 成果（数値 or 構造変化） |

**重要：** 数値がなくてもインパクトは成立します。  
「場当たり的なプロセスを再現可能なシステムに変換した」は正当な成果です。

## 初期段階の開発者へ

このフレームワークは次のものを必要としません：

- ❌ 何年もの実務経験
- ❌ 大規模チームプロジェクト
- ❌ すべてのプロジェクトの数値実績

必要なのはこれだけです：

- ✓ 実際の問題（自分自身の問題でも可）
- ✓ 実際に作ったもの
- ✓ 何が変わったかの誠実な説明

> 構造的な誠実さは、実績の誇張より常に強い。

## フレームワークの複製方法

1. このリポジトリをフォーク
2. プロジェクト内容を自分のものに置き換える
3. PSI変換を自分のプロジェクトに適用する
4. LinkedInプロフィールにこのリポジトリをリンクする
5. 投稿サイクルを開始する

> フレームワークが製品であり、キャリアシグナルがその出力です。
